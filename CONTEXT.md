# NutrAI 도메인 용어집

## 사용자 (User)
건강 관리 또는 체중 조절이 목표인 일반 성인. 당뇨·고혈압 등 특정 건강 상태를 가진 사람도 포함.

## 끼니 (Meal)
사용자가 하루 중 섭취하는 단위 식사. **5종**으로 분류:

| 종류 | 설명 |
|------|------|
| 아침 | 오전 식사 |
| 점심 | 낮 식사 |
| 간식 | 오후 가벼운 섭취 |
| 저녁 | 저녁 식사 |
| 야식 | 늦은 밤 섭취 |

음식 추가 시 시간대 기반으로 끼니를 자동 추천하되, 사용자가 직접 변경할 수 있다.

## 음식 (MealFood)
끼니를 구성하는 개별 식품. 이름·칼로리·탄수화물·단백질·지방 정보를 포함.

## 식단 기록 (MealRecord)
특정 끼니에 섭취한 음식들의 묶음. 끼니 종류와 시각을 포함.

- 앱 SQLite에 저장 (`meal`, `meal_food` 테이블)
- 서버는 stateless — 식단 이력은 요청 본문으로 전달받는다

## 건강 목표 (Goal)
사용자가 설정한 식단 관리 방향. `다이어트`, `체중 유지`, `근육 증진`, `건강 관리` 4종.

## 알레르기 (Allergy)
사용자가 기피해야 할 식품 성분. 11개 카테고리: 유제품, 견과류, 갑각류, 밀, 글루텐, 계란, 대두, 복숭아, 토마토, 고등어, 조개류.

## 알레르겐 카테고리 (Allergen Category)

사용자가 자신의 알레르기를 선택할 때 사용하는 분류 단위.
예: `유제품`, `견과류`, `갑각류`, `밀`, `글루텐`, `계란`, `대두`, `복숭아`, `토마토`, `고등어`, `조개류`.

- 사용자 프로필에 쉼표 구분 문자열로 저장된다: `"유제품,견과류"`
- 설정 화면에서 체크박스로 선택한다

## 알레르겐 키워드 (Allergen Keyword)

알레르겐 카테고리에 매핑된 실제 식재료 또는 음식명 키워드.
음식명 문자열 포함 여부(substring match)로 알레르기 위험을 판별한다.
예: `유제품` → `["우유", "치즈", "버터", "요거트", ...]`

- 단일 소스: `ai/allergens.py` — 서버(RAG, 추천 파이프라인)와 앱(AllergenService fallback)이 모두 이를 기준으로 한다
- 앱은 시작 시 `GET /allergens`로 최신 목록을 받아 메모리에 캐시하고, 오프라인이면 동일한 내용의 hardcoded fallback을 사용한다

## 알레르겐 감지 (Allergen Detection)

음식명 또는 RAG 검색 결과에서 사용자의 알레르겐 카테고리에 해당하는 키워드를 포함하는지 검사하는 연산.

검사가 이루어지는 세 지점:
1. **ChromaDB 검색 필터** (`rag_pipeline.py`) — 검색 결과에서 알레르겐 포함 문서 제거
2. **추천 후처리** (`recommendation_pipeline.py`) — LLM 응답의 추천 항목 검증
3. **앱 UI** (`AllergenService.detect`) — 음식 추가 화면에서 경고 배지·배너 표시

## 영양 정보 (Nutrition Info)

음식 100g 기준의 칼로리·탄수화물·단백질·지방·나트륨·식이섬유·당류.

- 저장소: ChromaDB 벡터 DB (`ai/rag_engine/chroma_db/`)
- 원본 데이터: 식품의약품안전처 CSV (`data/nutrition_db/`)

## 건강 상태 (Condition)
사용자의 기저 질환 또는 건강 상태 (예: 당뇨, 고혈압). AI 추천 시 제약 조건으로 활용.

## RAG 추천 파이프라인 (RAG Recommendation Pipeline)

사용자 메시지 + 프로필 + 식단 이력을 받아, ChromaDB에서 관련 영양 정보를 검색하고 Ollama LLM이 맞춤 식단을 생성하는 흐름.

단계: 다중 쿼리 생성 → 벡터 검색 → 알레르겐 필터 → LLM 생성 → 후처리(칼로리 검증, 알레르겐 재검증)
